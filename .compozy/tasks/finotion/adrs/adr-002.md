# ADR-002: iOS 17 Minimum Target and SwiftData for Local Persistence

## Status

Accepted

## Date

2026-05-25

## Context

The app requires local persistence for four data types: recurring payment definitions, monthly category budget goals, merchant aliases, and the offline queue of pending Notion writes. These must sync across reinstalls via iCloud. Two storage approaches are viable: CoreData + NSPersistentCloudKitContainer (iOS 16+) or SwiftData (iOS 17+). The iOS minimum target determines which is available.

## Decision

Set iOS 17 as the minimum deployment target and use SwiftData with CloudKit sync for all persistent models. Field mapping configuration (small, rarely changed) is stored as a Codable struct in NSUbiquitousKeyValueStore (iCloud KV Store). The Notion OAuth token is stored in the iCloud Keychain (`kSecAttrSynchronizable = true`).

## Alternatives Considered

### Alternative: iOS 16 minimum with CoreData

- **Description**: CoreData + NSPersistentCloudKitContainer for persistence; @ObservableObject for state management.
- **Pros**: Covers devices on iOS 16 that have not upgraded; CoreData is mature and well-documented.
- **Cons**: iOS 16 â†’ iOS 17 upgrade rate is very high (>90% of active devices are on iOS 17+); CoreData requires significantly more boilerplate than SwiftData; @ObservableObject is more verbose than the @Observable macro.
- **Why rejected**: The gain in coverage (<10% of devices) does not justify the additional development complexity. SwiftData's cleaner API reduces implementation time meaningfully.

## Consequences

### Positive

- SwiftData @Model classes require no boilerplate (no `@NSManaged`, no `fetchRequest()`, no `NSManagedObjectContext` threading management).
- `@Observable` macro provides automatic observation without manually publishing properties.
- CloudKit sync is configured declaratively at container initialization â€” no manual merge policy configuration.
- NSUbiquitousKeyValueStore is the right fit for FieldMapping: it syncs without requiring a CloudKit schema, has no conflict resolution complexity, and is capped at 1 MB (more than enough for a field mapping config).

### Negative

- Excludes users on iOS 16. Acceptable for a personal tool targeting a single user.
- SwiftData is newer than CoreData and has known edge cases in complex migration scenarios â€” but initial schema is simple enough that migrations are low-risk.

### Risks

- SwiftData + CloudKit sync can have propagation delays on first install after iCloud restore. Mitigation: show a loading state on first launch if sync is in progress.

## Implementation Notes

- `ModelConfiguration` is initialized with `cloudKitContainerIdentifier` pointing to the app's CloudKit container.
- All four SwiftData models (`RecurringPayment`, `BudgetGoal`, `MerchantAlias`, `PendingEntry`) are included in the same container.
- FieldMapping is stored via `NSUbiquitousKeyValueStore.default` under the key `"fieldMapping"`, encoded as JSON.
- Keychain entries use `kSecAttrAccessibleAfterFirstUnlock` + `kSecAttrSynchronizable = true`.
