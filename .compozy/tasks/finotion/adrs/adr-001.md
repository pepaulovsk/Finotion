# ADR-001: Product Approach — Notion Finance Companion

## Status

Accepted

## Date

2026-05-25

## Context

Finotion is a personal iOS finance tool for a single user who already has an organized Notion database for tracking expenses. Three distinct product approaches were evaluated: a lightweight Notion companion, a full-featured personal finance hub, and a notification-gateway-first minimal app. The choice determines scope, identity, and development timeline for the MVP.

## Decision

Finotion will be a **Notion Finance Companion**: a lightweight iOS app whose primary job is zero-friction expense capture (via NFC + Apple Shortcuts and Nubank notification parsing) paired with a contextual dashboard showing month-at-a-glance financials and local category budget goals. Notion remains the single source of truth for all transaction data. The app does not replicate or replace Notion — it extends it.

## Alternatives Considered

### Alternative B: Personal Finance Hub

- **Description**: Full-featured budgeting app where Notion is just the storage layer, comparable to YNAB or Copilot in feature depth (installments, multiple accounts, expense splitting, projections).
- **Pros**: More complete product long-term; stronger standalone value proposition.
- **Cons**: Significantly larger MVP scope (months of additional development); dilutes the "zero friction / Notion-first" identity; many features duplicate what Notion already provides.
- **Why rejected**: Scope and timeline are not justified for a personal tool. The Notion-companion identity is a clearer, more achievable differentiator.

### Alternative C: Notification Gateway

- **Description**: Capture-first minimal app. The primary UI is the notification and widget layer; the app itself is settings + a basic transaction list.
- **Pros**: Fastest MVP; laser focus on automated capture.
- **Cons**: Eliminates the dashboard, category goals, and recurring payment management — features the user explicitly requested.
- **Why rejected**: Misaligns with user requirements for in-app financial visibility and budget tracking.

## Consequences

### Positive

- Clear, narrow MVP scope with a realistic development timeline.
- Every feature decision is filtered through "does this reduce friction or improve financial context?" — keeps the product focused.
- Notion handles complex querying, filtering, and data export; Finotion doesn't need to replicate those.
- Local category goals keep data model simple without polluting the user's Notion workspace.

### Negative

- Advanced analytics (multi-month projections, trend regression) are not in scope — users who want deeper analysis must open Notion directly.
- The app is tightly coupled to Notion; if the user ever migrates away from Notion, the app loses its backend.

### Risks

- **Notion API rate limits**: heavy dashboard refresh could hit limits. Mitigation: cache responses locally and refresh only on user action or app foreground.
- **Nubank notification format changes**: if Nubank changes its push notification format, the regex parser breaks. Mitigation: design the parser to fail gracefully (surface raw notification for manual entry) and version the parsing rules.

## Implementation Notes

- All financial data persists in the user's existing Notion database — the app never creates a parallel data store for transactions.
- Category budget goals are stored in CoreData on-device, keyed by (category, year-month). They auto-carry forward each month but can be adjusted at any time.
- The dashboard fetches data from Notion on open and on pull-to-refresh; no background polling for dashboard data.
