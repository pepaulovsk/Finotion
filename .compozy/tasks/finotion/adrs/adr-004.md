# ADR-004: Offline Write Queue with Automatic Retry

## Status

Accepted

## Date

2026-05-25

## Context

The app's primary write operation — posting a transaction to Notion — will fail if the device is offline or if the Notion API is unavailable. The PRD requires the user to see a confirmation after saving (within 10 seconds of tapping NFC). Three failure strategies were considered: immediate error with manual retry, fail-silent with local save, or a persistent queue with automatic retry when connectivity returns.

## Decision

Implement a `PendingEntry` queue stored in SwiftData. When a Notion write fails, the transaction is saved as a `PendingEntry` with status `.pending`. A `Network.NWPathMonitor` observer triggers automatic retry whenever connectivity is restored. Successful writes update the entry status to `.synced` and are pruned after 24 hours. Entries that fail after 5 retries are marked `.failed` and surfaced in the UI.

## Alternatives Considered

### Alternative: Immediate error with manual retry

- **Description**: Show an error alert on failure; user taps "Try again" to retry.
- **Pros**: Simplest implementation; no local queue to manage.
- **Cons**: User must be online at the moment of registration. Loses data if they dismiss the error. Breaks the "under 10 seconds" success criteria when retrying manually.
- **Why rejected**: The PRD explicitly mentions queueing failed entries locally as a risk mitigation for Notion API downtime.

### Alternative: Error shown, no retry

- **Description**: Show an error, let the user re-enter the transaction manually.
- **Pros**: Zero implementation cost.
- **Cons**: Data loss; unacceptable for a finance tracking tool.
- **Why rejected**: Data integrity is non-negotiable.

## Consequences

### Positive

- Users see an optimistic confirmation immediately after tapping "Save" — the entry is persisted locally before the Notion call even completes.
- Connectivity restoration triggers a silent background sync — no user action required.
- SwiftData + CloudKit sync means the pending queue itself survives app restarts.

### Negative

- The Notion database may temporarily be out of sync with what the user sees in the app. The dashboard must clearly indicate when entries are "pending sync" vs. confirmed in Notion.
- Deduplication becomes important: if the app retries a write that actually succeeded (but returned a network timeout), it must not create a duplicate entry. Mitigation: use an idempotency key (UUID per PendingEntry) and check Notion for an existing entry with that key before re-posting.

### Risks

- Notion API does not have a native idempotency key header. Deduplication relies on querying Notion for an existing entry with the same `pendingId` stored in the Description/Notes field. This adds one extra API call per retry.

## Implementation Notes

```swift
@Model
class PendingEntry {
    @Attribute(.unique) var id: UUID
    var transactionData: Data      // JSON-encoded Transaction
    var createdAt: Date
    var retryCount: Int
    var lastAttemptAt: Date?
    var status: String             // "pending" | "synced" | "failed"
}
```

- `SyncService` observes `NWPathMonitor.pathUpdateHandler`. On `.satisfied` path, flushes all `.pending` entries.
- Retry back-off: immediate on first connectivity restore, then 30s / 2m / 10m / 30m intervals.
- Max retries: 5. After 5, status becomes `.failed` and a non-blocking UI indicator appears.
- The `pendingId` is stored in the Notion entry's Description field as a hidden marker (e.g., `[pid:UUID]`) for deduplication checks.
