# ADR-005: NotionService Protocol for API Abstraction

## Status

Accepted

## Date

2026-05-25

## Context

All Notion API interactions (creating transactions, querying history, fetching database schemas, creating template databases) must be encapsulated in a testable layer. The choice is between a protocol-backed service with URLSession, a third-party Swift Notion SDK, or direct URLSession calls in ViewModels.

## Decision

Define a `NotionService` protocol covering all Notion API operations used by the app. Provide two implementations: `LiveNotionService` (URLSession-backed, used in production) and `MockNotionService` (in-memory, used in tests and previews). All ViewModels and services depend on the protocol, never the concrete type.

## Alternatives Considered

### Alternative: Third-party Notion Swift SDK

- **Description**: Use an open-source Swift SDK for the Notion API (e.g., `notion-swift` on GitHub).
- **Pros**: Reduces boilerplate for serializing/deserializing Notion's JSON format.
- **Cons**: No official Notion-maintained Swift SDK exists; open-source alternatives are unmaintained or have limited coverage of the API; adds an external dependency that could break on Notion API updates.
- **Why rejected**: Maintaining control over the Notion API integration is safer for a long-lived personal tool. The Notion API surface used by Finotion is small enough that a bespoke client is low-effort.

### Alternative: Direct URLSession in ViewModels

- **Description**: Call URLSession directly from ViewModels without a service layer.
- **Pros**: Minimal files; fastest to write initially.
- **Cons**: Untestable without network access; violates MVVM's separation of concerns; duplicates request construction across ViewModels.
- **Why rejected**: Testability is required for the deduplication logic, field mapping, and offline queue — all of which need unit tests without hitting the real API.

## Consequences

### Positive

- All Notion API call construction, response parsing, and error mapping lives in one place.
- `MockNotionService` enables full unit test coverage and SwiftUI previews without a Notion account.
- Adding new API methods (e.g., for Phase 2 income tracking) requires only a protocol extension and a new implementation — no changes to ViewModels.
- Rate limiting and retry logic live in `LiveNotionService`, not scattered across callers.

### Negative

- Requires maintaining two implementations in parallel (live + mock). The mock must be kept up-to-date as the protocol evolves.

### Risks

- Notion API response shape can change without notice. Mitigation: use `@dynamicMemberLookup` or optional decoding for all Notion-specific fields; fail gracefully on unexpected shapes.

## Implementation Notes

```swift
protocol NotionService: Sendable {
    func fetchDatabases() async throws -> [NotionDatabase]
    func fetchDatabaseProperties(_ id: String) async throws -> [NotionProperty]
    func createDatabase(parentPageId: String) async throws -> NotionDatabase
    func queryTransactions(databaseId: String, filter: NotionFilter?) async throws -> [Transaction]
    func createTransaction(_ tx: Transaction, databaseId: String) async throws -> String
    func addCategoryOption(_ name: String, databaseId: String, propertyId: String) async throws
}
```

- `LiveNotionService` is initialized with the access token and `FieldMapping`. All requests inject the token as `Authorization: Bearer {token}`.
- Base URL: `https://api.notion.com/v1/`. Notion-Version header: `2022-06-28`.
- Response decoding uses `JSONDecoder` with `keyDecodingStrategy = .convertFromSnakeCase`.
- All errors are mapped to a `NotionError` enum: `.unauthorized`, `.rateLimited`, `.serverError(Int)`, `.decodingError(Error)`, `.networkError(Error)`.
