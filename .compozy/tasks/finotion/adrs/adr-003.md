# ADR-003: MVVM with @Observable for App Architecture

## Status

Accepted

## Date

2026-05-25

## Context

The app is a greenfield SwiftUI project with several distinct features (onboarding, dashboard, expense entry, recurring payments, merchant aliases, settings). An architecture pattern must be chosen to organize view-model relationships, state management, and service injection. Options range from lightweight MVVM to structured unidirectional patterns (TCA) to coordinator-augmented MVVM.

## Decision

Use simple MVVM: each feature has a `View` and a corresponding `@Observable ViewModel`. Services (e.g., `NotionService`, `RecurringDispatchService`) are injected into ViewModels via the initializer. Navigation is handled with SwiftUI's native `NavigationStack` and sheet/fullScreenCover modifiers. No Coordinator or Router layer.

## Alternatives Considered

### Alternative: The Composable Architecture (TCA)

- **Description**: Unidirectional data flow pattern from Point-Free. State, actions, reducers, and effects are strictly separated.
- **Pros**: Excellent testability, predictable state mutations, strong community.
- **Cons**: Significant boilerplate; steep learning curve; best suited for teams, not solo personal tools. Adds a third-party dependency (`swift-composable-architecture`).
- **Why rejected**: The overhead is not justified for a personal app with a single developer. MVVM achieves equivalent testability for this scope through protocol-based service injection.

### Alternative: MVVM + Coordinator

- **Description**: Adds a Coordinator layer to manage navigation flows, decoupling views from routing logic.
- **Pros**: Cleaner separation of navigation from views in complex apps.
- **Cons**: Adds structural complexity that SwiftUI's `NavigationStack` already handles adequately.
- **Why rejected**: The app's navigation graph is shallow. `NavigationStack` with value-based routing is sufficient.

## Consequences

### Positive

- Each feature module is self-contained: `FeatureView.swift` + `FeatureViewModel.swift` + `FeatureService.swift`.
- `@Observable` eliminates `@Published` boilerplate and provides fine-grained view updates.
- Services are defined as protocols, enabling mock injection in tests without any additional framework.
- Minimal dependencies — no third-party architecture libraries needed.

### Negative

- No centralized state store; global state (e.g., the current field mapping, auth token) must be passed via environment or injected at the root level.
- Navigation logic lives in views, which can become verbose for deep flows.

### Risks

- ViewModels can grow large if business logic is not extracted into focused service types early. Mitigation: keep ViewModels as thin coordinators between views and services; push all business logic into services.

## Implementation Notes

- Root state (auth status, field mapping, selected database) is held in a single `AppState` @Observable class, injected via SwiftUI Environment.
- Services are created once at the app root and injected downward.
- Feature modules follow the directory structure: `Features/Dashboard/`, `Features/ExpenseEntry/`, `Features/Onboarding/`, etc.
